What Are Token Bound Accounts?

ERC-6551 gives every NFT its own smart contract wallet. Each TAO Punk gets a deterministic Ethereum address that can hold TAO, receive payments, and execute transactions — controlled exclusively by whoever owns the punk.

The punk IS the wallet. When you own a punk, you control its wallet. When you sell or transfer the punk, the wallet (and everything inside it) transfers with it.

Each TBA is an ERC-1167 minimal proxy deployed via CREATE2, making its address deterministic and predictable. You can send TAO to a punk's wallet address before it's even deployed — the address is computed from the punk ID, chain ID, and NFT contract.

Why TBAs for TAO Punks?

Revenue Follows the NFT

When an AI agent earns fees through the Agent Registry (ERC-8004), the TAO goes directly into the punk's TBA. If the punk changes hands, the new owner controls all accumulated earnings. This creates a genuine revenue-bearing NFT — the punk's value includes its wallet balance.

No More Claiming

The old model required holders to call claim() to withdraw earned fees. With TBAs, fees land in the punk's wallet automatically on fulfillment. The holder uses execute() to move funds whenever they want.

Composable Identity

A punk's TBA can hold other tokens, interact with DeFi protocols, and serve as an on-chain identity. The punk becomes more than a profile picture — it's an autonomous economic agent on Bittensor.

Key Design Decisions

  • Counterfactual addresses: TAO can be sent to a punk's TBA before it's deployed. The address is deterministic via CREATE2.
  • CALL only: TBA restricts to operation type 0 (CALL). No DELEGATECALL or CREATE for security.
  • No migration cost: The registry hasn't been deployed yet — clean integration, no state to preserve.
  • Permissionless TBA creation: Anyone can call createPunkTBA(). It's idempotent.
  • NonReentrant: fulfill() and batchFulfill() are protected since they now make external calls.

Security Model

Only the punk owner controls the TBA. The execute() function checks TaoPunksV2.ownerOf(tokenId) dynamically — ownership is always current, never cached.

Access Control

  • execute() — Reverts with NotAuthorized if msg.sender != ownerOf(punkId)
  • Operation restriction — Only CALL (operation 0). No DELEGATECALL or CREATE.
  • Dynamic ownership — isValidSigner() always reads the current NFT owner
  • Isolation — Each punk's TBA is completely independent. Punk 1's owner cannot access Punk 2's TBA.

Reentrancy Protection

Since _fulfill() now makes an external .call{value}("") to the TBA, both fulfill() and batchFulfill() are protected with OpenZeppelin's nonReentrant modifier.

Escrow Invariant

Model Invariant
Old (Pull) balance >= sum(pendingWithdrawals) + sum(pending_fees)
New (TBA) balance >= sum(pending_fees)

TAO leaves the registry on fulfill, so the contract only needs to hold enough for pending (unfulfilled) query fees. This dramatically reduces escrow risk.

What Transfers With Your Punk

When a TAO Punk changes hands, the new owner gets everything:
Asset Description
The NFT ERC-721A token (TaoPunksV2)
The Wallet (TBA) Smart contract wallet with all TAO inside
Agent Identity ERC-8004 agent registration, metadata, query history
Future Revenue All subsequent query fees go to the punk's TBA
Agent Control Ability to activate, pause, set fees, update metadata

This makes TAO Punks revenue-bearing assets. A punk with an active AI agent and accumulated earnings is worth more than a freshly minted one. The market can price in:

  • Current TBA balance (liquid TAO)
  • Agent query volume and revenue history
  • Agent reputation and metadata
  • Future earning potential

Deployment Plan

  1. Deploy TaoPunkAccount (implementation contract, deployed once)
  2. Deploy ERC6551Registry (deployed once, creates all TBAs)
  3. Deploy TaoPunkAgentRegistry with TBA registry + implementation addresses
  4. TBAs are created on-demand via createPunkTBA() (permissionless, idempotent)
  5. Alternatively, TBAs can receive TAO at their deterministic addresses before creation
No migration needed. The Agent Registry has not been deployed to mainnet yet. This is a clean integration with zero existing state to preserve.
